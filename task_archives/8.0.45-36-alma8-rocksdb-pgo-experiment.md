# 8.0.45-36 AlmaLinux 8 RocksDB PGO Experiment

## Summary

- date: `2026-03-18`
- target: `Percona Server 8.0.45-36`
- platform class: `AlmaLinux 8 compatible`
- build option: `WITH_ROCKSDB=ON`
- objective: check whether dual-engine (`innodb` + `rocksdb`) PGO training reduces `InnoDB read_only` uplift compared with the no-RocksDB reference

## Result

Baseline (`WITH_ROCKSDB=ON`):

| db_engine | workload | TPS |
| --- | --- | ---: |
| innodb | point_select | 70502.46 |
| innodb | read_only | 3528.90 |
| rocksdb | point_select | 51188.65 |
| rocksdb | read_only | 2179.51 |

PGOed (`train=innodb rocksdb`, `validate=innodb rocksdb`):

| db_engine | workload | TPS | improvement |
| --- | --- | ---: | ---: |
| innodb | point_select | 110067.86 | 56.12% |
| innodb | read_only | 5036.29 | 42.72% |
| rocksdb | point_select | 81225.69 | 58.68% |
| rocksdb | read_only | 3534.88 | 62.19% |

## Conclusion

- Dual-engine RocksDB-enabled training did not reduce `InnoDB read_only` uplift in this setup.
- RocksDB itself also benefited strongly under the same training set.
- This run remained good enough that the fallback “InnoDB-only training” experiment was not needed.

## Notes

- A `coverage-mismatch` warning policy adjustment was required during the first `profile-use` rebuild.
- Dataset reuse remained important for disk safety, but internal host and path details are intentionally omitted from this public archive.
